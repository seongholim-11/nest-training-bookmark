# Markly 프로젝트 기술 정리 가이드

이 문서는 프로젝트 진행 중 습득한 주요 개념들을 정리한 가이드입니다.

---

## 🏗 1. NestJS: 건물을 짓는 설계도 (Architecture)
NestJS는 대규모 서버를 안정적으로 짓기 위한 '설계도(Framework)'입니다.

*   **Module (모듈)**: 프로젝트를 기능별로 나누는 '구역'입니다. (예: 주방, 거실, 화장실)
*   **Controller (컨트롤러)**: 외부 손님(클라이언트)의 요청을 받는 '안내데스크'입니다. "어떤 요청을 처리할지" 결정합니다.
*   **Service (서비스)**: 안내데스크 뒤에서 실제로 요리하거나 일을 하는 '실무자'입니다. 핵심 로직(비즈니스 로직)이 들어갑니다.
*   **Dependency Injection (의존성 주입)**: 실무자가 필요한 도구(DB 등)를 직접 사러 가지 않고, 관리자(NestJS)가 알아서 챙겨주는 방식입니다.

## 🗄 2. Prisma: 데이터베이스와의 통역사 (ORM)
컴퓨터 언어(TypeScript)와 데이터베이스 언어(SQL)는 서로 다릅니다. Prisma는 이 사이에서 통역사 역할을 합니다.

*   **Schema (스키마)**: 데이터가 어떤 모양으로 담길지 정하는 '틀'입니다. (User, Bookmark 등)
*   **Migration (마이그레이션)**: 스키마 설계도의 변경 이력을 기록하는 '버전 관리'입니다.
*   **Relationship (관계)**:
    *   **1:N (일대다)**: 한 명의 유저가 여러 개의 북마크를 가질 수 있습니다.
    *   **N:M (다대다)**: 하나의 북마크에 여러 태그가 붙고, 하나의 태그가 여러 북마크에 쓰일 수 있습니다. (우리는 `BookmarkTag`라는 중간 연결 고리를 만들었습니다.)

## 🔐 3. Security: 소중한 정보 지키기
사용자의 비밀번호를 보호하고 "당신이 맞는지" 확인하는 과정입니다.

*   **Bcrypt (암호 해싱)**: 비밀번호를 그대로 저장하면 위험합니다. Bcrypt는 비밀번호를 아무도 알아볼 수 없는 '암호문(Hash)'으로 바꿉니다. 한 번 바꾸면 원래 비밀번호로 되돌릴 수 없는 '일방향성'이 특징입니다.
*   **Hashing vs Encryption**: 
    *   **Hashing**: 복구 불가 (비밀번호 저장용)
    *   **Encryption**: 복구 가능 (중요 데이터 전송용)

## 📡 4. DTO & Validation: 통과할 수 있는 데이터인지 검사
클라이언트가 "비밀번호를 숫자 1개만 보낸다거나", "이메일 형식이 아닌 글자를 보내면" 서버가 고장 날 수 있습니다.

*   **DTO (Data Transfer Object)**: 데이터가 이동할 때 담는 '표준 바구니'입니다.
*   **class-validator**: 바구니에 담긴 물건이 상하지 않았는지(올바른 형식인지) 자동으로 검사합니다.

## 🛠 5. Modern Node.js의 도전: ESM vs CommonJS
우리가 겪었던 가장 큰 오류 중 하나였습니다.

*   **CommonJS (CJS)**: 예전부터 쓰던 방식 (`require`, `module.exports`).
*   **ES Modules (ESM)**: 최신 표준 방식 (`import`, `export`).
*   **문제 해결**: Prisma 7과 NestJS는 서로 다른 방식을 선호할 때가 있어, 우리는 `tsconfig.json` 설정을 통해 이들의 언어를 통일(CommonJS)시켜 충돌을 해결했습니다.

## 🎫 6. JWT: 서버가 주는 입장권 (Authentication)
로그인을 성공하면 서버는 사용자의 '디지털 신분증(Token)'을 발급합니다.

*   **Stateless (상태 없음)**: 서버는 사용자가 누군지 일일이 기억하지 않습니다. 사용자가 신분증(JWT)을 보여주면 그 안에 적힌 정보를 보고 믿어줍니다.
*   **Payload & sub**: JWT 안에는 'payload'라는 데이터 영역이 있습니다. 우리는 여기에 표준 규격인 `sub`(사용자 ID)와 이메일을 담아, 서버가 매번 DB를 조회하지 않아도 "이 토큰이 누구 것인지" 알 수 있게 했습니다.
*   **Access Token**: 짧은 유효 기간(15분)을 가진 실제 신분증입니다.

## 🔐 Phase 2: 인증 시스템 구현 상세 (Security)

사용자의 안전한 회원가입과 로그인을 위해 다음과 같은 고도화된 설정을 적용했습니다.

1.  **비밀번호 보호 (Bcrypt)**:
    - 회원가입 시 비밀번호를 `await bcrypt.hash(rawPassword, salt)`를 통해 암호화하여 저장합니다.
    - 로그인 시에는 암호화된 값을 복구하는 것이 아니라, `bcrypt.compare`를 통해 입력된 값과 암호화된 값을 대조합니다.

2.  **보안 에러 핸들링**:
    - "유저가 없음"과 "비밀번호 틀림"의 에러 메시지를 하나로 통일하여 사용자가 존재하는지 여부를 외부에서 알 수 없도록 보안을 강화했습니다.
    - NestJS의 `UnauthorizedException`을 사용하여 401 에러를 표준화된 포맷으로 반환합니다.

3.  **동적 JWT 설정 (registerAsync)**:
    - `ConfigService`를 활용해 환경 변수(`.env`)에서 시크릿 키와 만료 시간을 읽어와 런타임에 JWT를 설정하도록 구현했습니다.
    - `getOrThrow`를 사용하여 설정값이 누락되었을 때 서버가 즉시 에러를 발생시켜 잘못된 설정을 방지했습니다.

4.  **리프레시 토큰 (Refresh Token) 시스템**:
    - **UX와 보안의 균형**: 유효기간이 짧은 Access Token(15분)만 사용하면 사용자가 자주 로그인해야 하는 불편함이 있습니다. 이를 해결하기 위해 긴 유효기간(7일)을 가진 Refresh Token을 도입했습니다.
    - **토큰 로테이션 (Token Rotation)**: 리프레시 토큰을 사용할 때마다 새로운 리프레시 토큰을 재발급하여, 한 번 사용된 토큰은 폐기함으로써 보안성을 높였습니다.
    - **데이터베이스 기반 세션 제어**: 리프레시 토큰을 Bcrypt로 해싱하여 DB(`hashedRt`)에 저장합니다. 이를 통해 서버는 원격으로 특정 기기의 로그아웃을 강제하거나 토큰을 무효화할 수 있습니다.
    - **Logout(로그아웃)**: 사용자가 로그아웃하면 DB의 `hashedRt` 값을 삭제하여, 탈취된 토큰이 있더라도 더 이상 사용할 수 없도록 '서버 측 무효화'를 구현했습니다.

---

> **왜 이런 복잡한 과정을 거칠까요?**
> 단순히 기능을 만드는 것을 넘어, **보안(Security)**, **확장성(Scalability)**, **유지보수성(Maintainability)**이 좋은 '프로페셔널한' 코드를 만들기 위함입니다.
